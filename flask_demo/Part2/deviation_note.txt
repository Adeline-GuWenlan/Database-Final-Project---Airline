
This solution derives the relational schema from the posted conceptual model in the Part 1 answer key and implements the **seat classes** anti‑automation challenge.  Each airplane’s seats are represented by the `SeatClass` table, which replaces the `seat_capacity` attribute in the original `Airplane` entity.  The `Ticket` table includes a `price_charged` attribute and references `SeatClass` to indicate the class reserved.  Tickets are linked to flights via a simple foreign key rather than the more general “marketing flight” relation since code‑share flights were not selected.

Some relationships in the conceptual model are implemented by embedding the foreign key in the participating entity rather than creating a separate relation.  For example, the ownership of airplanes by airlines and the assignment of flights to airplanes and airports are represented through foreign keys in `Airplane` and `Flight`, respectively.  The multivalued `permission` attribute of airline staff is modeled using a separate `StaffPermission` table.  The `purchases` relation stores a single record per ticket–customer pair; the optional booking agent is stored as a nullable foreign key.  These choices maintain data integrity while simplifying the schema and align with the relational design principles exemplified in the Part 1 answer key.
